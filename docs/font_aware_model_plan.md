# 字体选择功能下的模型模块推进计划

更新时间：2026-07-09  
适用项目：“一笔成章”模型模块  
背景：软件端新增字体/书家风格选择功能，例如“赵孟頫楷体”等。模型侧需要从单纯的通用笔画分割，逐步升级为“通用分割 + 风格条件化评价/反馈”。

## 1. 总体判断

字体选择功能会改变模型的最终目标，但不应该打断 7 月当前主体。

原计划中的六通道分割任务仍然成立：

- `vec1`
- `vec2`
- `vec3`
- `vec4`
- `vec5`
- `keypoint`

这些通道描述的是笔画方向、交叉和关键点结构，属于相对通用的底层视觉能力。无论用户选择赵孟頫、欧阳询、颜真卿还是柳公权，模型仍然需要先稳定识别字形结构、笔画区域和关键点。

因此，新的模型路线应该分成两层：

1. 通用结构层：负责分割、关键点、笔画结构抽取。
2. 风格评价层：根据用户选择的目标字体/书家风格，判断当前书写与目标风格的差异，并生成更具体的反馈。

一句话策略：

> 7 月先把“看懂字”的模型做稳；8 月以后再让模型“按某个书家的标准看字”。

## 2. 为什么不建议现在直接训练“每个字体一个模型”

新增字体选择后，最直觉的方案是给每种字体单独训练一个模型，但这不适合作为第一版路线。

主要原因：

- 当前数据量偏小，每个字体/书家单独切分后，样本会进一步变少。
- 不同书家风格之间既有差异，也有大量共享结构；完全分开训练会浪费数据。
- 软件后续可能继续新增字体，如果每新增一种字体就重新训练一个完整模型，维护成本会很高。
- 当前 7 月目标是完成高精度主体模型，过早引入字体条件会增加变量，导致基线难以判断。

所以第一阶段不做“赵孟頫模型”“颜真卿模型”这种完全独立模型，而是做：

> 一个通用分割主干 + 可扩展的风格条件模块。

## 3. 新功能下的模型任务拆分

字体选择功能加入后，模型模块可以拆成三个任务。

### 3.1 任务 A：通用六通道分割

这是当前 7 月主线，保持不变。

输入：

- RGB 汉字图像
- 原始尺寸

输出：

- `[H, W, 6]` 概率图
- 六张二值 mask
- 各通道阈值
- 推理耗时

目标：

- 稳定识别笔画方向、交叉、端点和关键点。
- 作为后续风格评价的结构基础。

### 3.2 任务 B：字体/书家风格条件化评价

这是软件新增字体选择后真正新增的能力。

输入需要扩展为：

- RGB 汉字图像
- 原始尺寸
- 当前汉字 ID，例如 `永`、`人`、`書`
- 目标风格 ID，例如 `zhao_mengfu_kaishu`
- 可选目标参考图，例如该书家该字的标准字图

输出可以扩展为：

- 全局风格相似度评分
- 各笔画风格评分
- 结构偏差，例如重心、宽高比、部件间距
- 笔画偏差，例如粗细、长短、角度、起收笔位置
- 可视化差异图
- 面向前端/大模型反馈模块的结构化诊断结果

### 3.3 任务 C：风格参考库与对齐

字体选择功能本质上需要“目标标准”。仅靠学生手写图像是不够的，还需要每种目标风格的参考。

需要建立：

- 字符集合：项目支持哪些字。
- 风格集合：支持哪些书家/字体。
- 标准图：每个“字符 × 风格”的参考图。
- 元数据：书家、字体、碑帖来源、书体、版本、是否人工校准。
- 可选标注：参考字的六通道 mask、关键点、骨架、笔画序列。

这部分更像模型和内容资产之间的桥梁，建议由模型组和美术/书法成员共同维护。

## 4. 当前 7 月计划如何调整

### 4.1 不改变 7 月主线

7 月仍然优先完成：

- 数据审计
- `manifest.csv`
- 固定 `train/val/test` 划分
- U-Net 重测基线
- SegFormer-B2 高精度主线
- 六通道分割指标
- 单图推理接口

原因是：字体评价也依赖稳定的分割结果。如果底层结构都分不准，后面的风格反馈会变成空中楼阁。

### 4.2 7 月立即预留的改动

虽然不训练风格模型，但数据 schema 现在就应该预留字段。

建议在后续 manifest 中增加以下可选列：

| 字段 | 含义 |
| --- | --- |
| `char_id` | 当前汉字 ID |
| `writer_id` | 书写者 ID |
| `source_id` | 数据来源 ID |
| `target_style_id` | 用户选择或样本对应的目标风格 |
| `script_type` | 书体，如楷书、行书、隶书 |
| `reference_id` | 对应标准字图 ID |
| `reference_image_path` | 标准字图路径 |
| `style_source` | 风格来源，如碑帖、字帖、字体文件、人工整理 |
| `style_label_quality` | 风格标签可信度 |

其中 `target_style_id` 建议使用稳定英文/拼音 ID，不要直接依赖中文显示名。

示例：

```text
zhao_mengfu_kaishu
ouyang_xun_kaishu
yan_zhenqing_kaishu
liu_gongquan_kaishu
```

前端可以显示“赵孟頫楷体”，但模型和接口内部使用 `zhao_mengfu_kaishu`。

## 5. 8 月以后的模型升级路线

### 5.1 第一阶段：风格参考库

先不要急着训练模型，先把目标风格的参考资产建起来。

每种目标风格至少需要：

- 覆盖软件常用字。
- 每个字一张标准图。
- 尽量统一尺寸、边距、黑白阈值。
- 记录来源。
- 由书法/美术成员人工确认质量。

建议优先支持 2–3 种风格，而不是一口气支持很多字体。

第一批推荐：

- 赵孟頫楷体
- 欧阳询楷体
- 颜真卿楷体

如果团队数据压力较大，也可以先只做一种风格，把完整链路跑通。

### 5.2 第二阶段：基于规则和几何特征的风格评价

在深度风格模型之前，先做一个可解释的几何评价 baseline。

基于六通道 mask 和关键点，可以计算：

- 字形外接框比例
- 重心位置
- 笔画长度
- 笔画角度
- 笔画粗细
- 起笔/收笔位置
- 交叉点位置
- 左右/上下结构比例
- 与参考图的 Chamfer 距离或骨架距离

这个阶段的优点是：

- 不依赖大量风格标注。
- 很适合生成前端反馈。
- 失败原因容易解释。
- 可以作为后续神经网络风格评价模型的 baseline。

### 5.3 第三阶段：风格条件化模型

当有了参考库和一定数量的人工评价数据后，再训练风格条件模型。

推荐结构：

```text
学生书写图像
    ↓
通用分割主干 SegFormer
    ↓
六通道结构特征
    ↓
风格条件模块 ← target_style_id / 参考字图
    ↓
风格评分、笔画级偏差、反馈特征
```

风格条件模块可以有两种实现。

#### 方案 A：Style ID Embedding

输入一个 `target_style_id`，模型学习每种风格的嵌入向量。

优点：

- 实现简单。
- 推理快。
- 适合风格数量较少且固定的情况。

缺点：

- 新增风格需要再训练或微调。
- 对没有见过的风格泛化较弱。

#### 方案 B：Reference Encoder

输入目标风格的标准字图，让模型自己编码参考字。

优点：

- 更适合软件端“选择字体/字帖”的形态。
- 新增风格时，只要提供参考图，模型理论上更容易适配。
- 便于做“当前字 vs 目标字”的差异可视化。

缺点：

- 工程复杂度更高。
- 需要解决学生字和参考字的对齐问题。

建议路线：

> 先做方案 A 作为第一版风格条件模型；同时保留 Reference Encoder 的接口，为后续扩展做准备。

## 6. 数据采集与标注策略调整

新增字体选择后，后续采样不能只问“这个字写得好不好”，还要问“相对于哪个风格好不好”。

### 6.1 样本元数据必须记录目标风格

每个样本至少记录：

- 字符
- 书写者
- 采集时间
- 目标风格
- 书体
- 设备/背景
- 是否临摹
- 对应参考图

### 6.2 人工复核方式调整

高怡然或书法专业成员复核时，不再只看分割是否正确，还要增加风格维度：

- 是否像目标书家风格
- 哪些笔画最不像
- 是结构问题还是笔画形态问题
- 是否存在系统性误判

可以先使用三级标签：

```text
0 = 明显不符合目标风格
1 = 部分符合，但有明显问题
2 = 基本符合目标风格
```

不要一开始就做 100 分制，因为主观噪声会更大。

## 7. 推理接口升级建议

当前 7 月接口：

```text
输入：RGB 图片及原始尺寸
输出：[H,W,6] 概率图、六张二值掩码、各通道阈值、推理耗时
```

后续字体选择版接口建议升级为：

```json
{
  "image": "RGB image",
  "original_size": [H, W],
  "char_id": "yong",
  "target_style_id": "zhao_mengfu_kaishu",
  "reference_id": "zhao_mengfu_kaishu_yong_v1"
}
```

输出：

```json
{
  "schema_version": 2,
  "channels": ["vec1", "vec2", "vec3", "vec4", "vec5", "keypoint"],
  "probability_map": "[H,W,6]",
  "binary_masks": "[H,W,6]",
  "thresholds": {
    "vec1": 0.5,
    "vec2": 0.5,
    "vec3": 0.5,
    "vec4": 0.5,
    "vec5": 0.5,
    "keypoint": 0.45
  },
  "target_style_id": "zhao_mengfu_kaishu",
  "style_scores": {
    "global": 0.82,
    "structure": 0.78,
    "stroke_shape": 0.85,
    "keypoint_position": 0.80
  },
  "stroke_feedback": [],
  "latency_ms": 0
}
```

注意：`schema_version=1` 保持当前分割接口；`schema_version=2` 再加入字体/风格相关输出。

## 8. 验收标准调整

### 8.1 7 月不变

7 月模型主体仍按原验收：

- 五方向 Macro Dice 相比重测 U-Net 提升至少 3 个百分点。
- 关键点 F1 提升至少 5 个百分点。
- 困难集 Macro Dice 提升至少 5 个百分点。
- 训练、评测、单图推理能单命令运行。

### 8.2 字体选择功能的新增验收

后续新增：

- 同一个输入字，选择不同目标风格时，反馈结果应有合理差异。
- 风格评分与人工复核结果 Spearman 相关系数达到可接受水平。
- 每种目标风格至少有独立测试集。
- 新增字体/风格时，不需要重写主模型训练框架。
- 风格模块失败时，六通道分割结果不受影响。

## 9. 阶段性排期建议

### 7 月 9–31 日

继续完成原模型主体：

- U-Net 重测基线
- SegFormer-B2 主线
- 困难集
- 单图推理
- 实验报告

同时轻量预留：

- 在文档和配置中明确 `target_style_id`
- 后续 manifest 支持风格字段
- 与前端约定不要把中文字体名直接写死进模型接口

### 8 月上旬

建立第一版风格参考库：

- 选择 1–3 种目标风格
- 每种风格整理常用字参考图
- 建立 `style_registry.yaml`
- 建立 `reference_manifest.csv`

### 8 月中旬

做几何特征 baseline：

- 分割结果与参考图对齐
- 输出结构差异和笔画差异
- 给前端提供第一版风格化反馈

### 8 月下旬

开始风格条件模型：

- Style ID Embedding 版本
- 少量人工风格评分数据
- 多任务训练：分割 + 风格评分
- 与几何 baseline 对比

## 10. 当前最该做的决策

为了避免软件和模型接口后面反复改，建议团队尽快确认三个问题：

1. 第一版软件到底支持哪些字体/书家风格？
2. 用户选择的是“书家风格”，还是“具体字体文件”？
3. 每个风格是否能提供每个汉字的标准参考图？

我的建议是：

> 第一版不要把范围铺太大。先选择 1 个主风格，例如赵孟頫楷体，把“选择风格 → 获取参考 → 分割 → 对齐 → 反馈”的链路跑通，再扩展到其他风格。

这样模型研发不会被数据规模拖垮，软件功能也能较快做出可演示版本。
